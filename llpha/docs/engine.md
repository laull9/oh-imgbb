# 🧠 Llpha 核心工作流爬虫引擎设计文档（v1）

---

# 1. 🎯 设计目标

Llpha 的核心不是“爬取网页”，而是构建一个：

> **以网页为输入，以任务流为核心的可扩展数据处理引擎**

它的目标是：

* 将网页解析结果转化为“可执行任务”
* 支持任务递归生成（self-expanding workflow）
* 支持多类型数据采集（HTML / 图片 / API / 文件）
* 支持插件扩展解析与决策逻辑
* 支持高并发、可控调度与失败恢复

---

# 2. 🧩 核心抽象模型

整个系统只围绕四个核心抽象：

---

## 2.1 Task（任务）

> 系统的最小执行单元

任务不仅是“请求一个 URL”，而是一个**可演化的执行意图**

包含：

* 目标资源（URL / API / 文件）
* 执行类型（抓取 / 提取 / 下载 / 跟踪）
* 上下文信息（来源、深度、状态）
* 扩展元数据（插件使用、规则参数）

👉 本质：**“行为描述 + 上下文携带体”**

---

## 2.2 Page（页面结果）

> Task 执行后的标准化输出

包含：

* 原始响应（HTML / JSON / binary）
* 解析后的 DOM 表示
* 元信息（状态码、header、mime）
* 可供分析的结构化视图

👉 本质：**“统一的中间数据表达层”**

---

## 2.3 Action（行为抽象）

> 从 Page 中提取出的“下一步行为意图”

Action 是系统“智能扩展能力”的核心来源，包括：

* 数据提取行为
* 链接跳转行为
* 文件下载行为
* 新任务生成行为
* 插件触发行为

👉 本质：**“从信息到行为的转换结果”**

---

## 2.4 Task Graph（任务图）

> 系统运行的全局结构

任务之间不只是队列关系，而是：

* 依赖关系
* 扩展关系
* 分支关系
* 递归关系

形成一个动态生成的有向图：

👉 本质：**“不断生长的执行图（Execution Graph）”**

---

# 3. 🔁 核心执行模型（工作流引擎）

Llpha 的运行不是 pipeline，而是：

> **事件驱动 + 图扩展 + 反馈闭环**

---

## 3.1 主循环模型

系统持续执行以下闭环：

1. 调度器选择任务
2. 执行任务获取 Page
3. 解析 Page 生成 Action
4. Action 转换为新 Task
5. 新 Task 回流调度器
6. 图结构持续扩展

---

## 3.2 执行本质

系统行为等价于：

> **“不断消费任务，并生成新的任务空间”**

---

## 3.3 关键特征

* 非线性（Graph-based）
* 非静态（动态生成）
* 可回溯（任务链）
* 可中断（状态可恢复）

---

# 4. ⚙️ 核心模块设计

---

## 4.1 Scheduler（调度器）

职责：

* 管理任务生命周期
* 控制执行顺序
* 处理优先级
* 维护任务状态

关键能力：

* 队列管理（FIFO + priority）
* 去重机制
* 深度控制（防递归爆炸）
* 并发额度控制（限制运行中任务数量）
* 失败重试策略

👉 本质：**“任务流控制器”**

---

## 4.2 Fetcher（执行层）

职责：

* 执行 HTTP 请求
* 管理连接池
* 处理代理与重试
* 返回标准 Page

关键能力：

* 并发控制
* 请求策略（header / proxy / retry）
* 失败恢复
* 性能优化

👉 本质：**“网络执行引擎”**

---

## 4.3 Parser（解析层）

职责：

* 将原始响应转为结构化 DOM
* 提供统一查询能力（CSS / XPath）

关键能力：

* HTML 标准化
* DOM tree 构建
* 数据预处理

👉 本质：**“语义化转换器”**

---

## 4.4 Task Generator（核心决策引擎）

> 系统最核心模块（“智能来源”）

职责：

* 从 Page + Action 推导新 Task
* 决定是否继续扩展
* 控制任务递归逻辑

关键能力：

* 链接发现
* 条件判断
* 任务拆解
* 数据驱动扩展

👉 本质：
**“把信息变成可执行未来的系统大脑”**

---

## 4.5 Plugin System（扩展系统）

职责：

* 注入自定义解析逻辑
* 扩展 Action 生成规则
* 定义领域爬虫能力（电商 / 图片 / API）

关键能力：

* Hook机制（page / task / action）
* DSL扩展
* 可插拔策略

👉 本质：**“领域能力注入层”**

---

# 5. 🔄 数据流模型

系统中所有数据流动遵循：

---

## 5.1 单向基础流

```
Task → Fetch → Page → Parse → Action → Task
```

---

## 5.2 闭环扩展流

```
Task Graph
   ↑       ↓
Action → New Task
```

---

## 5.3 多分支扩展

一个 Page 可产生：

* 数据提取任务
* 链接追踪任务
* 下载任务
* API调用任务

👉 即：

> **1 → N 扩展模型**

---

# 6. 🧠 核心设计原则

---

## 6.1 Task-first（任务优先）

系统所有逻辑围绕 Task 展开，而不是 URL 或 HTML。

---

## 6.2 Graph-driven（图驱动）

执行路径不是线性的，而是可扩展图结构。

---

## 6.3 Stateless execution（执行无状态）

执行层不保存业务状态，只消费 Task。

---

## 6.4 Declarative expansion（声明式扩展）

Page 不直接控制流程，而是通过 Action 描述未来行为。

---

## 6.5 Plugin-isolation（插件隔离）

插件只负责“规则”，不参与执行调度。

---

# 7. ⚡ 关键系统特性

---

## 7.1 可递归扩展

* 支持无限层级爬取
* 可控制 depth / scope

---

## 7.2 可中断恢复

* Task 状态持久化
* 可恢复 Graph 执行

---

## 7.3 可扩展任务类型

支持：

* HTML爬取
* API调用
* 图片下载
* 自定义 DSL

---

## 7.4 可插件化领域能力

例如：

* 电商解析插件
* 新闻聚合插件
* API抓取插件

---

# 8. 🧭 系统本质总结

Llpha 的核心不是 scraper，而是：

> # 🧠 一个“任务驱动的递归工作流引擎（Task-driven Execution Graph Engine）”

它本质上等价于：

* Scrapy（但更抽象）
* Temporal（但面向爬虫）
* Airflow（但实时递归）
* Playwright（但不是浏览器）

---

# 9. 🚀 一句话架构定义

> Llpha 是一个基于 Task Graph 的递归式网页计算引擎，将网页转化为可扩展的任务流，并通过插件系统实现领域化数据抽取与自动化扩展。
